# UI技術設計：Solar Gravity Simulator

対象: UI実装（TypeScript + ブラウザ）  
参照: requirements-ui.md, requirements-physics.md

---

## 1. 目的

UI仕様（画面仕様）を実装可能な形に落とし込む。  
この文書は厳密仕様の固定ではなく、実装で確定した内容を追記していく運用を前提とする。

---

## 2. 設計方針

- UIは「学習体験」を優先し、操作から結果までのフィードバックを短くする
- 判定ロジック（成功/失敗）は物理側、UIは表示と遷移反映に徹する
- 右パネルの常時情報と上部の一時通知を分離する
- MVPはPCブラウザ優先、スマホは最低限の操作保証
- 初期値やしきい値などの可変要素はuiConfigへ集約し、実装時に調整可能とする
- 「実装時に決定」とした項目は、値を設定可能な作りで実装する

---

## 3. 技術選定

### 3.1 フロントエンド基盤

- 言語: TypeScript
- ビルド/開発サーバ: Vite
- 描画: Canvas 2D API
- 物理連携: Rust/WASM（wasm-bindgen）

### 3.2 UI実装スタイル

- UI構築: React
- レイアウト: CSS Gridを基本に、スマホ時は1カラムへ切り替え
- 状態管理: React Context + useReducer（必要時のみ軽量ストアを追加）
- 時間倍率UIは仕様書の定義（requirements-ui.md 3.3）に準拠
- 実装時決定の可変値はUI設定オブジェクト（uiConfig）へ集約
- uiConfigはTypeScriptの定数ファイルで管理し、外部JSON/YAMLはMVPでは採用しない
- MVPではセッション内状態のみ扱い、ページ再読み込み時に初期化する
- localStorageによる設定上書きは将来拡張として扱う

### 3.3 品質管理

- 開発規約は仕様書の定義（requirements-ui.md 12-1）に準拠
- 数値表示は小数桁を丸め、可読性を優先する

### 3.4 実装環境と配置

フロントエンド実装は `src/frontend/` をルートとする：

```
src/
├─ frontend/             ← フロントエンド実装（React + Vite）
│  ├─ src/
│  ├─ public/
│  ├─ mock/              ← WASM モック（開発用）
│  ├─ package.json
│  ├─ vite.config.ts
│  ├─ tsconfig.json
│  └─ index.html
└─ wasm/                 ← 本番WASM（将来連携時用）
```

運用方針：
- フロントエンド開発は `src/frontend/` をルートとして進める
- `src/frontend/package.json` で Node.js 依存関係を管理
- `src/frontend/mock/` にWASMモック実装（本番コードと明確に分離）
- Node.jsはnvmで管理し、Node 22 LTSをプロジェクトルートの `.nvmrc` で固定
- `npm run dev` / `npm run build` は `src/frontend/` で実行
- 将来WASM連携時、`src/wasm/` のビルド成果物を `src/frontend/public/` に配置

## 4. UI構成案

### 4.1 画面分割

- 基本領域構成は仕様書の定義（requirements-ui.md 2.1）に準拠
- 操作パネルは下部固定
- 右パネルはPC常時表示、狭幅時は折りたたみ

### 4.2 状態遷移（UI）

- 状態遷移は仕様書の定義（requirements-ui.md 8）に準拠
- UIは状態遷移の表示責務のみを持つ
- ミッション結果は右パネル内に表示
- ミッション終了後の結果表示は、数秒後に通常表示へ自動復帰
- 軌道ガイド/天体トレイル初期値は両方ON
- スマホ最低保証はパン＋ズーム

---

## 5. コンポーネント分割（案）

- `SimulationCanvas`: 天体/ロケット/軌跡の描画、カメラ操作
- `ToastMessageBar`: 上部一時通知
- `ControlPanel`: 発射入力、操作ボタン、時間倍率
- `InfoPanel`: ミッション情報、テレメトリー、結果表示
- `UiStore`: 画面状態、選択ミッション、入力値、表示トグル
- `WasmGateway`: WASM APIとの入出力変換

---

## 6. データ連携方針（UI <- WASM）

- 1フレームごとにスナップショット取得
- UIで扱う最低データ:
  - 天体配列（位置、半径、色、強調フラグ）
  - ロケット状態（位置、速度、軌跡、状態）
  - テレメトリー（速度、高度、重力加速度、最近傍天体）
  - ミッション状態（進行中、成功、失敗、終了理由）

注記:
- 厳密なフィールド定義は接続実装時に確定
- WASM API の型定義はモック実装と同一インターフェースで管理

## 7. ユースケース

各ユースケースは、開始条件・ユーザー操作・期待されるUI結果の3点で定義する。

### UC-01 ミッション選択して発射

- 開始条件: Idle、操作パネルが表示されている
- ユーザー操作: ミッション選択後に角度と初速度を設定し、発射を押す
- 期待されるUI結果: 状態がRunningへ遷移し、上部に発射通知、右パネルでテレメトリー更新開始

### UC-02 飛行中に時間倍率を変更

- 開始条件: Running
- ユーザー操作: 時間倍率をpause/1x/2x/4xで切り替える
- 期待されるUI結果: シミュレーション進行速度が切り替わり、停止時は描画更新のみ継続

### UC-03 飛行中にカメラを操作

- 開始条件: RunningまたはIdle
- ユーザー操作: パン、ズームを行う
- 期待されるUI結果: カメラが追従し、注目対象の観察が継続できる

### UC-04 成功時の結果確認

- 開始条件: Runningで成功条件成立
- ユーザー操作: 右パネルの結果を確認する
- 期待されるUI結果: FinishedSuccessへ遷移し、成功結果が表示され、数秒後に通常表示へ自動復帰

### UC-05 失敗時の結果確認とリセット

- 開始条件: Runningで失敗条件成立
- ユーザー操作: 右パネルの結果を確認し、リセットを押す
- 期待されるUI結果: FinishedFailed表示後、リセットでIdleへ戻り入力可能状態に復帰

### UC-06 表示トグルの切り替え

- 開始条件: IdleまたはRunning
- ユーザー操作: 軌道ガイド表示、天体トレイル表示を切り替える
- 期待されるUI結果: Canvas上の補助表示が即時反映される

## 8. 開発フロー

実装ルートは `src/frontend/` で、以下の手順で開発を進める：

1. `src/frontend/` で `nvm use` により Node 22 LTS を有効化
2. `src/frontend/` で `npm install` によりパッケージをインストール
3. `src/frontend/` で `npm run dev` により Vite 開発サーバを起動
4. モックは `src/frontend/mock/` 内に TypeScript で実装
5. UI が固まった後、WASM API 仕様を確定し、モックをWASM連携へ置換
